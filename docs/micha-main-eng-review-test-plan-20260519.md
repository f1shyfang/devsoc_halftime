## Affected Pages/Routes

- `/` — bounty feed (read-only, no automated tests; manual eyeball during Saturday checkpoint)
- `/b/[slug]` — mobile join + check-in landing; handles `?ci=<token>` for Check-in QR
- `/room/[slug]` — projector view with Realtime + 3s polling fallback
- `/host/[slug]` — admin curation queue (admin token gated)

## Key Interactions to Verify (E2E `tests/e2e/demo-flow.spec.ts`)

Playwright multi-context test simulating the live pitch demo:

1. **Judge context:** scans Discover QR → lands on `/b/halftime-tabledrop` → fills tag form (name, 3 tags) → submits → page shows "awaiting_curation" state
2. **Host context:** opens `/host/halftime-tabledrop` with admin token → sees new pending request appear via Realtime within 1s → taps `[Accept]`
3. **Judge context (Realtime-driven transition):** `/b/halftime-tabledrop` auto-transitions to "accepted, scan check-in QR" state within 1s
4. **Judge context:** navigates to `/b/halftime-tabledrop?ci=<test_token>` → `check_in` RPC fires → page shows "checked in" confirmation
5. **Projector context (third browser context):** `/room/halftime-tabledrop` shows judge's name + tags within 2s of check-in

Assertions: end-to-end latency under 2s per Realtime hop, no duplicate rows on projector, no console errors.

## Edge Cases (RPC unit tests, `tests/rpc/check_in.test.ts`)

`check_in` state machine — one test per input state:

- `invalid_token` — call `check_in('halftime-tabledrop', 'wrong-token')` → returns `{ok: false, reason: 'invalid_token'}`, no `room_presence` row mutated
- `bounty_closed` — bounty with `ends_at < now()` → returns `{ok: false, reason: 'bounty_closed'}`
- `not_requested` — no `bounty_requests` row for `auth.uid()` → returns `{ok: false, reason: 'not_requested'}` (NO auto-accept — verifies curation premise is preserved)
- `awaiting_curation` — `bounty_requests.status = 'pending'` → returns `{ok: false, reason: 'awaiting_curation'}`
- `rejected` — `bounty_requests.status = 'rejected'` → returns `{ok: false, reason: 'rejected'}`
- `accepted (happy)` — `bounty_requests.status = 'accepted'` → returns `{ok: true, reason: 'checked_in'}` AND upserts `room_presence` with `is_present = true`
- `idempotent re-scan` — call `check_in` twice in succession → second call also returns `{ok: true}`, no duplicate `room_presence` row, no error

`request_to_join` (additional unit tests):

- Duplicate request: same `auth.uid()` calling twice → returns same `request_id`, no new row
- `bounty_full`: when `count(accepted requests) >= max_seats` → raises `bounty_full`
- `bounty_closed`: when `now() > ends_at` → raises `bounty_closed`

## Critical Paths

1. **Discover → request → accept → check-in → projector update** (the demo). E2E covers this.
2. **`check_in` state machine** — all 6 input states. Unit tests cover this.

## NOT in test scope (Reviewer Concerns)

- Realtime fan-out load test (20+ concurrent subscribers) — moved to Friday spike (1B decision)
- Polling fallback path — manual verification during Saturday 4pm checkpoint
- `accept_request` race condition with concurrent accepts → `max_seats` exceeded — captured as TODO, fix is `FOR UPDATE` clause
- Mobile viewport rendering of post-accept screen — manual verification
- Visual regression of `/room/[slug]` projector — manual verification


Both run locally as `npm run test:e2e` and `npm run test:rpc`. Neither blocks CI for the hackathon timeline — they're tools for the dev team's confidence, not deploy gates. Wire them as required PR checks post-Halftime.
