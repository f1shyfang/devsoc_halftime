# Supabase ‚Üí Neon Migration ‚Äî Design

**Date:** 2026-05-30
**Status:** Approved (design)
**Branch:** `feat/neon-migration`

## Goal

Fully cut the app off Supabase and onto **Neon Postgres**, deployed on **Vercel**.
Neon is a plain Postgres connection string ‚Äî it provides none of the Supabase
platform services (PostgREST data API, RPC-over-HTTP, Realtime, Storage, Edge
Functions). So this is not just a database move; it is a rewrite of the data,
realtime, and storage layers of the app.

Auth is **not** in scope: the app already replaced Supabase Auth with a
device-bound UUID cookie (`quest_device_id`, set by `proxy.ts`) in migration
`20260519000006_remove_auth_use_device_id.sql`. Every `SECURITY DEFINER`
function already takes `p_user_id uuid` explicitly, so the functions are
auth-context-free and portable as-is.

## Non-goals

- No change to game logic or UX.
- No reintroduction of auth.
- No rewrite of the Postgres functions' internals ‚Äî they move verbatim.
- The `.claude/worktrees/*` copies are out of scope (separate experiments).

## Target architecture

The core shift: the browser stops talking to a backend-as-a-service directly.
Today `supabase-js` runs in client components and hits PostgREST / Realtime /
Storage. After the cutover, **all data access is server-side only**:

```
Before:  Browser ‚îÄ‚îÄ(supabase-js)‚îÄ‚îÄ‚ñ∫ Supabase [PostgREST | Realtime | Storage | Edge fn]
After:   Browser ‚îÄ‚îÄ(fetch/SWR)‚îÄ‚îÄ‚ñ∫ Next.js API routes ‚îÄ‚îÄ(Drizzle)‚îÄ‚îÄ‚ñ∫ Neon Postgres
                                          ‚îî‚îÄ(@vercel/blob)‚îÄ‚îÄ‚ñ∫ Vercel Blob (photos, cards)
```

## Decisions (confirmed with user)

| Area | Decision |
|---|---|
| Data layer | Drizzle ORM over Neon; **keep existing SQL functions** and call them via raw SQL |
| Driver | `drizzle-orm/neon-http` (single-statement RPC/reads); escalate a path to `neon-serverless` (WS pool) only if it needs a multi-statement transaction |
| Realtime | SWR polling of read-only API routes (~2‚Äì3s) |
| Storage | Vercel Blob (`@vercel/blob`) |
| Data | Real production data preserved via `pg_dump --data-only` |
| RLS | **Dropped.** Authz enforced by `SECURITY DEFINER` functions (`p_user_id` checks) + API routes scoping reads by the device-id cookie |
| Deploy | Vercel |

## Components

### 1. Database layer ‚Äî `lib/db/`

- `lib/db/client.ts` ‚Äî Drizzle client over `drizzle-orm/neon-http`, reads
  `DATABASE_URL` (Neon pooled connection string).
- `lib/db/schema.ts` ‚Äî generated by `drizzle-kit pull` (introspection) after the
  initial Neon load. No hand-written DDL ‚Üí no drift. Used for the
  `.from().select()` / `.insert()` reads we are replacing.
- **Postgres functions move verbatim.** App code calls them with, e.g.:
  ```ts
  await db.execute(
    sql`select * from quest_create_team(${userId}, ${huntId}, ${teamName})`
  );
  ```
  Functions in scope (from `database.types.ts`, all take `p_user_id` first):
  `quest_create_team`, `quest_ensure_profile`, `quest_join_team`,
  `quest_apply_penalty`, `quest_start_hunt`, `quest_unlock_clue`,
  `mvp_join_game`, `mvp_get_player_state`, `mvp_confirm_start`, `mvp_ack_walk`,
  `mvp_submit_anagram`, `mvp_submit_geocache_code`, `mvp_record_photo`,
  `mvp_complete_player`.
- `createAdminClient()` consumers (`app/api/rooms/free/route.ts`,
  `scripts/enrich-buildings.ts`) move to the Drizzle/Neon client.

### 2. Schema + data migration

1. `pg_dump --schema-only --schema=public` from Supabase ‚Üí clean out
   Supabase-isms (RLS policies, `auth.*` / `storage.*` refs, role GRANTs, the
   storage-bucket migrations) ‚Üí load into Neon.
2. `pg_dump --data-only --schema=public` from Supabase ‚Üí restore into Neon
   (preserves real sessions/teams/players rows). Disable triggers during load.
3. `drizzle-kit pull` against Neon ‚Üí generate `schema.ts`.
4. Going forward: Drizzle owns table DDL; functions/triggers live in a
   version-controlled `db/functions.sql` applied by a small migrate script.
   The old `supabase/migrations/` are retired but kept for history.

### 3. Realtime ‚Üí SWR polling

Replace the three `.channel().subscribe()` sites with SWR polling of read-only
routes (`refreshInterval` ~2‚Äì3s):

- `app/quest/demo/[huntSlug]/standings/standings-view.tsx`
  ‚Üí `GET /api/quest/standings/[huntId]`
- `app/leaderboard/leaderboard-view.tsx`
  ‚Üí `GET /api/mvp/games/[gameId]/leaderboard`
- `app/quest/demo/[huntSlug]/play/play-shell.tsx` session sync
  ‚Üí `GET /api/quest/sessions/[id]`; the "navigate to finale on session update"
  logic keys off polled state instead of a realtime event.

### 4. Storage ‚Üí Vercel Blob

- New `POST /api/uploads/quest-photo` receives the file server-side, calls
  `put()` from `@vercel/blob`, returns the public URL. Replaces
  `supabase.storage.from('quest-photos').upload()/getPublicUrl()` in
  `play-shell.tsx` and `mvp-play-shell.tsx`.
- `BLOB_READ_WRITE_TOKEN` env var.

### 5. Edge function ‚Üí Next.js route

`supabase/functions/generate_results_card` (Deno) rewritten as
`POST /api/quest/results-card` (Node / Vercel Function), writing the generated
image to Vercel Blob. Image-generation logic ported faithfully (internals to be
read during planning).

### 6. Env vars

- **Remove:** `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`,
  `SUPABASE_SERVICE_ROLE_KEY`.
- **Add:** `DATABASE_URL` (Neon), `BLOB_READ_WRITE_TOKEN` (Vercel Blob).
- Keep: `FOURSQUARE_API_KEY` and the optional overrides.
- Update `.env.example`.

### 7. Dependencies

- **Remove:** `@supabase/ssr`, `@supabase/supabase-js`.
- **Add:** `drizzle-orm`, `@neondatabase/serverless`, `@vercel/blob`, `swr`;
  dev: `drizzle-kit`.

## Error handling

- API routes return structured JSON errors (`{ error: string }`) with
  appropriate status codes; client call sites map them to the existing UI error
  states (the current `{ data, error }` destructuring patterns are preserved at
  the fetch boundary).
- Functions that `raise exception 'p_user_id is required'` surface as 400s.
- Polling routes are read-only and idempotent; transient errors are retried by
  SWR without user-facing disruption.

## Testing

- `route.test.ts` (currently mocks supabase) rewritten to mock the new DB layer.
- Each new API route gets a unit test (happy path + auth/validation failure).
- End-to-end manual verification of full flows before ripping out supabase:
  quest team create/join ‚Üí start ‚Üí unlock clue ‚Üí penalty ‚Üí finale ‚Üí results
  card; MVP join ‚Üí play ‚Üí leaderboard.

## Rollout phases (for the implementation plan)

1. Provision Neon; migrate schema + data; generate Drizzle schema; stand up
   `lib/db`. App still on Supabase.
2. Build read/RPC API routes; cut over one feature end-to-end (quest team flow)
   as the proof.
3. Migrate remaining RPC/read sites; swap realtime ‚Üí polling.
4. Storage ‚Üí Blob + results-card route.
5. Rip out supabase deps; fix tests; verify full flows; update env/docs.

## Risks / open items

- **Results-card port:** the Deno edge function's image generation may use
  Deno-specific APIs; the Node port must reproduce output. Read internals during
  planning.
- **Connection limits:** Neon HTTP driver is stateless and serverless-friendly;
  confirm the pooled connection string is used to avoid exhausting connections.
- **Data load ordering:** FK constraints require correct table order or deferred
  constraints during `--data-only` restore.
