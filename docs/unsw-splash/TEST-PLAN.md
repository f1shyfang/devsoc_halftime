## Affected Pages/Routes

- `/join?event=halftime` — Mobile onboard. Display-name entry, anonymous auth, idempotent join, late-joiner splice (post D3).
- `/play` — Main player loop. Map + target dot + SPLASH button + Realtime + Wake Lock + audio prime + (in finale) QR scanner.
- `/room/halftime` — Projector view. Anonymized heatmap + ticker + finale finalist roster.
- `/admin` — Host queue. Token gate + initial graph + undo + mark dead + trigger finale.

## Key Interactions to Verify

### Door check-in flow (`/join`)
- Scan Discover QR card, page loads, anonymous auth succeeds.
- Enter `display_name` 1-30 chars; submit; row appears in `players` table with `checked_in=false`.
- Submit duplicate name (someone already entered "Sam") → suffix "Sam (2)" auto-applied and shown to user.
- Edge-case operator on `/admin` marks `checked_in=true`; player sees no UI change (host-side).
- Late joiner (post-10:30am, `games.status='live'`) → join succeeds and player is immediately spliced into the cycle; existing hunter's `/play` retargets via Realtime within 5s.

### Active player loop (`/play`)
- Wake Lock acquired on mount; screen stays on while page visible.
- Page hidden (app switch, lock screen) → Wake Lock released.
- Page visible again → `visibilitychange` handler re-acquires Wake Lock; "You were out for X min" banner if gap > 2 min.
- Audio prime: first user gesture on `/play` primes the splash WAV; subsequent fires play reliably on iOS Safari.
- Heartbeat every 5s: `locations` row updates; map dot moves on hunter's screen via Realtime within ~5s.
- Distance > 15m → SPLASH button shows "MOVE CLOSER — Xm" (gray, locked).
- Distance < 15m for 5s → button progresses through "Unlocking" countdown → "SPLASH" (blue, pulsing).
- Stale GPS (no recent location) → splash RPC returns `no_recent_location`; UI shows clear error, not generic "MOVE CLOSER".
- WS disconnect > 5s → 1s polling fallback engages; banner shown.
- Hunter SPLASH tap → both phones get flash + sound + vibrate (centralized via D9 `playSplashEffect`).
- Target sees full-screen obituary + button to `/room` spectator view.

### Splash event (catharsis moment — load-bearing)
- iOS Safari: audio plays on target's phone (D7 prime path).
- Android Chrome: audio plays on hunter and target.
- Vibrate fires on phones with `navigator.vibrate` support.
- Flash overlay covers full viewport for 1s.
- `/room` projector ticker shows event (anonymized, last 5 events only — Reviewer Concern #10).

### Finale flow
- Host taps `BEGIN FINALE` on `/admin` → `games.status='finale'`, all alive players marked `finalist`, bib_number + bib_qr_token assigned.
- Player `/play` switches: map removed, target reveal panel shown (target name + avatar color + "HUNT THEM" CTA).
- Camera permission requested on scanner open; if denied → fallback messaging ("Allow camera in browser settings to splash").
- Hunter scans target's bib QR → `splash_finale(token)` succeeds → flash + sound + vibrate on both phones.
- Hunter scans non-target bib → `wrong_target` error; no state change; no info leaked about scanned player.
- Hunter scans invalid/corrupted QR → `invalid_qr` error.
- `/room` shows finalist roster + 60s countdown + ticker.

### Admin operations (`/admin`)
- Token gate: missing/wrong token → 401; correct token → access.
- Initial graph shuffle: builds cycle from `checked_in=true` players; sets `games.status='live'`; randomized per-call.
- Undo splash: target restored alive; hunter's prior assignment restored; post-splash assignment deleted.
- Mark dead (for ghost-alive): player marked dead; their hunter retargeted to player's old target.
- Trigger finale: as above.

## Edge Cases

### Splash RPC state machine (CRITICAL — load-bearing)
- No active assignment → `not_playing`.
- Hunter is dead (D1 added check) → `already_dead_self`. **REGRESSION-RISK**.
- Target is dead (lost the race) → `already_dead`. **REGRESSION-RISK**.
- Game in finale mode → `finale_use_scan`.
- Hunter or target has no recent location → `no_recent_location`.
- Distance > 15m → `out_of_range`.
- Happy 3+alive: hunter inherits target.old_target as new target.
- 2-alive paradox (target.old_target == hunter) → set `games.status='finale'`, skip new assignment.
- Concurrent splashes A→B and B→C: row locks serialize; both succeed in some order; cycle remains coherent.
- Concurrent `admin_undo_splash` with new splash on target: assignment lock prevents stale write.
- Late joiner spliced mid-splash transaction: serialized via assignment FOR UPDATE lock.

### Distance calculation (pure unit test)
- Haversine at boundary: 14.9m → in range; 15.1m → out; exactly 15.0m → choose one (recommend in-range, design says ≤15m).
- Antipodal: 20015km → far out of range, no overflow.
- Same point: 0m → in range.
- Realistic UNSW campus extremes: Library to Mathews ~400m → out.

### Join game
- Empty display_name → 400.
- 31-char display_name → 400.
- Unicode/emoji display_name → accept (trim only).
- Control chars (\\n, \\t, NUL) → strip or reject.
- Reserved names ("Admin", "Host") → reject or auto-suffix? Decide.
- Duplicate auth.uid() → idempotent upsert; no error.

### Wake Lock + visibilitychange
- Lock acquired on first user gesture.
- Lock released when page hidden (verify by checking `wakeLock.released`).
- Lock re-acquired on visibilitychange → 'visible'.
- Banner shown if gap > 2 min.
- Multiple background/foreground cycles: no leaked sentinels.

### Realtime / WS
- WS disconnect > 5s → polling fallback engages; data continues to refresh every 1s.
- WS reconnect → polling stops; subscriptions resume.
- Server returns stale data during polling: `locations.updated_at` used to detect.

### Late joiner splice (D3)
- Joins at 11:00am with games.status='live' → spliced; appears in cycle within one Realtime tick on previous tail's `/play`.
- Two late joiners join simultaneously → both spliced sequentially via FOR UPDATE locks; cycle still coherent.
- Joins after `games.status='finale'` → rejected.

### Camera + QR
- Permission denied on first scanner open → fallback UI ("Allow camera in browser settings").
- Permission granted later → scanner re-initialized.
- Bib QR partially obscured → @zxing/browser tolerates ~25% occlusion; fully obscured → no scan.
- Two finalists' bibs in frame → scanner picks first decoded; deterministic per @zxing behavior.

### Privacy
- Player A queries `players` table directly: only own row + their hunter + their target + all finalists (post-finale-trigger).
- Player A queries `locations`: only own + hunter's + target's.
- Public `/room` queries `splashes`: visible but no names rendered until finale.
- After finale: all finalist names revealed.

## Critical Paths (must work end-to-end)

1. **Door-to-active**: scan QR → enter name → host marks checked_in → 10:30 graph assigned → `/play` shows target.
2. **First splash of the day**: hunter walks within 15m → 5s hold → tap → both phones flash+sound+vibrate → target sees obituary → ticker on `/room` updates.
3. **Mid-game ghost-alive**: player phone screen off 10 min → host gets ping on `/admin` → at 20 min `admin_mark_dead` fires → orphaned hunter retargets within 5s.
4. **Finale trigger**: host taps BEGIN FINALE at ~6 alive → all phones get reveal banner → bibs distributed → 60s countdown → scans land → final survivor crowned → `/room` displays winner.
5. **Network blip recovery**: hunter walks into stairwell → WS drops → polling banner appears → hunter exits stairwell → WS reconnects → state coherent.

## Test framework recommendation

- **vitest** for unit/integration tests (RPC logic against `supabase start` local Postgres).
- **Playwright** for one E2E spec covering the happy splash flow (two browser contexts simulating two phones).
- **No LLM evals needed** — design uses no LLM/prompt paths.

## CI

- `npm test` runs vitest + Playwright.
- Pre-commit hook: vitest only (Playwright too slow for pre-commit).
- GitHub Actions: full suite on PR to main.
