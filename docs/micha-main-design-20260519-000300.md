## Problem Statement

Sydney lacks a way for like-minded people to find each other for spontaneous, in-person, time-bound encounters. The status quo is invite-only group chats, broadcast platforms (Lu.ma, Partiful, Eventbrite) that optimize for events rather than connection, or dating apps that optimize for 1:1 romantic matching. None of them solve the *"I want a small curated table of interesting people, here, soon"* problem.

User's framing: *"OH no I have no space or way to connect with likeminded ppl about HACKATHON IDEAS or whatever."*

DevSoc Halftime is the demo venue and the first proof point. The product applies to any third space where curated spontaneous companionship is the unit of value.

## What Makes This Cool

The product is inspired by the **mechanic** of rent-a-gf (bookable person-anchored time-bound companionship) with the transaction stripped. What remains is a primitive that doesn't really exist in Western app ecosystems: *"I want company who matches X, here, now"* answered by curated people.

The Halftime demo plan is the strongest single thing about this project: the pitch IS the product running live. Tables pushed in, chairs brought, props out, team seated and discussing the idea loudly, projector showing `/room/halftime-tabledrop` filling with judges who scanned the QR and got curated into the room. The room IS the demo.

Codex named the actual product: *"you are not excited by bounties. You are excited by turning coordination into a social ritual."* That's the identity. **TableDrop** — summon a table.

## Constraints

- 3-5 devs, 48-72h hackathon window
- Stack: Next.js 15 (App Router) + Supabase (SSR auth + Realtime) + Tailwind + shadcn/ui
- Mobile-first joiners, desktop projector
- Hackathon venue wifi unreliable under load — must have fallbacks
- Halftime venue hardcoded for the demo; product generalizes post-Halftime

## Premises

1. **Wedge:** Bounty-for-connection. Rent-a-gf mechanic, transaction stripped. Distinct from Lu.ma (broadcast), Meetup (recurring), Tinder (dating), Eventbrite (ticketed). Verb: *summon*.
2. **Curation is product, not feature.** Soho House energy. The moat.
3. **Demo north star:** Live room snapshot, framed as *"this bounty summoned these people."*
4. **Payoff = connection itself, no payment v1.** Cash replaced by curation + intentionality + check-in gate.
5. **Halftime is the demo venue, NOT the product scope.**

Codex challenged premise #4 → check-in gate adopted (joiners appear on snapshot only after physical in-venue scan).

## Approach A — Full Spec

### Pages (4)

```
/                    Bounty feed. 3 seeded bounties, Halftime pinned at top.
/b/[slug]            Mobile join + check-in landing. Handles ?ci=<token>.
/room/[slug]         Projector view. Live snapshot, check-in-gated. RT + polling fallback.
/host/[slug]         Curation queue. Accept/reject + mark-arrived. Auth-gated.

# CUT FROM V1: /bounties/new, /me  (post-Halftime)
```

### Data Model

```sql
profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  name text not null,
  avatar_url text,
  tags text[] not null default '{}',     -- captured at request-to-join
  inserted_at timestamptz default now()
);

bounties (
  id uuid primary key default gen_random_uuid(),
  slug text unique not null,
  host_id uuid not null references profiles(id),
  title text not null,
  description text,
  location text,
  venue_lat numeric, venue_lng numeric,
  starts_at timestamptz not null,
  ends_at timestamptz not null,
  max_seats int not null,
  checkin_token text not null,            -- random opaque token in Check-in QR
  -- Bounty is "open" while now() between starts_at and ends_at.
  -- No status enum; compute on read. Removes need for a close-button.
  created_at timestamptz default now()
);

bounty_requests (
  id uuid primary key default gen_random_uuid(),
  bounty_id uuid not null references bounties(id),
  profile_id uuid not null references profiles(id),
  status text not null default 'pending',  -- 'pending' | 'accepted' | 'rejected'
  created_at timestamptz default now(),
  unique (bounty_id, profile_id)
);

room_presence (
  id uuid primary key default gen_random_uuid(),
  bounty_id uuid not null references bounties(id),
  profile_id uuid not null references profiles(id),
  checked_in_at timestamptz,
  last_seen_at timestamptz,
  is_present boolean not null default false,
  unique (bounty_id, profile_id)
);

-- Realtime publication (CRITICAL — both tables required):
--   ALTER PUBLICATION supabase_realtime ADD TABLE room_presence;
--   ALTER PUBLICATION supabase_realtime ADD TABLE bounty_requests;
--   ALTER TABLE room_presence REPLICA IDENTITY FULL;
--   ALTER TABLE bounty_requests REPLICA IDENTITY FULL;
--
-- Without bounty_requests in the publication, the joiner's /b/[slug] page
-- cannot detect that the host accepted their request (Step 3 → 4 of the
-- demo flow silently breaks).
```

### Auth Model (joiners + host, explicit)

- **Host:** Supabase magic-link auth (email link). Logged in once Friday on the demo laptop. Session persists in cookies via `@supabase/ssr` (already wired in template). `auth.uid()` returns the host's id inside RPCs.
- **Joiners (judges, scanners):** **Supabase anonymous auth** (`supabase.auth.signInAnonymously()` on first visit to `/b/[slug]`). Creates an `auth.users` row with no email, returns a JWT, persists in cookie. `auth.uid()` is non-null inside `request_to_join` and `check_in`. No login UX shown to the judge. The anon JWT is what RLS policies key off (`auth.uid() = profile_id` for own-row reads).
- A `profiles` row is created/upserted by `request_to_join` keyed off `auth.uid()`. The host's profile row is created/upserted at Friday-evening magic-link login (one-time bootstrap step in the build plan).

### Auth (Supabase Magic Link, one-shot Friday pre-auth) — revised

- Host signs in via existing `app/auth/login` magic link flow once on Friday.
- Supabase Auth session persists in cookies (handled by `@supabase/ssr`, already wired in this template).
- `/host/[slug]` is protected via the existing proxy middleware (`proxy.ts` → `updateSession`) which redirects unauthenticated requests to `/auth/login`.
- During the pitch: NO re-auth. Cookies already valid (Supabase sessions last 60 days by default). Test Friday evening.
- This uses native Supabase auth — `auth.uid()` is available inside SECURITY DEFINER RPCs for host verification. NO custom fingerprint code.

Why not magic link as v1 production auth? Because for the demo we need ONE host pre-authed on ONE device. For post-Halftime hosts, the same magic link flow extends naturally — no rewrite required.

### SECURITY DEFINER RPCs (explicit contracts)

```sql
-- Called by joiner from /b/[slug] when they hit "request to join"
function request_to_join(p_bounty_slug text, p_name text, p_tags text[])
returns table (request_id uuid, status text);
-- Behavior:
--   Upserts a profile row for auth.uid() with p_name + p_tags.
--   Inserts a bounty_requests row in 'pending'.
--   Returns the request id + 'pending'.
--   If duplicate (auth.uid() already has a row for this bounty): returns existing row.
--   If bounty not open (ends_at < now()): raise 'bounty_closed'.
--   If bounty full (max_seats already accepted): raise 'bounty_full'.

-- Called by host from /host/[slug]
function accept_request(p_request_id uuid) returns void;
-- Behavior:
--   Verifies caller's auth.uid() == bounties.host_id for this request's bounty.
--   Sets bounty_requests.status = 'accepted'.
--   Upserts a room_presence row (is_present = false, ready for check-in).
--   If bounty_full: raise 'bounty_full'.

function reject_request(p_request_id uuid) returns void;
-- Behavior:
--   Verifies caller is the host.
--   Sets status = 'rejected'.

-- Called by joiner from /b/[slug] when they scan Check-in QR
function check_in(p_bounty_slug text, p_token text) returns table (ok boolean, reason text);
-- Behavior (explicit state machine):
--   If bounties.checkin_token != p_token: return (false, 'invalid_token').
--   If now() not between starts_at and ends_at: return (false, 'bounty_closed').
--   Look up bounty_requests row for (bounty_id, auth.uid()).
--   If no row: return (false, 'not_requested').  -- explicitly NO auto-create. Preserves curation premise.
--   If status = 'pending': return (false, 'awaiting_curation').  -- explicitly NO auto-accept. Host must accept first.
--   If status = 'rejected': return (false, 'rejected').
--   If status = 'accepted':
--     Upsert room_presence row, set is_present = true, checked_in_at = now().
--     Return (true, 'checked_in').

-- Optional host override for accessibility cases (no phone, can't scan)
function host_mark_arrived(p_request_id uuid) returns void;
-- Behavior:
--   Verifies caller is the host.
--   Sets room_presence.is_present = true for that request's profile.
```

The `check_in` contract is the spec review's biggest fix — it now explicitly REJECTS auto-accept for pending/rejected/no-row states, preserving premise #2 (curation is product, not feature). The Check-in QR is taped to the projector base in plain sight; without curation gating, anyone in the audience could scan and bypass the host.

### RLS Policies

- `room_presence` SELECT: `is_present = true` for anyone (anon-key projector reads).
- `room_presence` INSERT/UPDATE: blocked at table level; only mutated via RPCs above (SECURITY DEFINER bypasses RLS).
- `bounty_requests` SELECT: own row (`auth.uid() = profile_id`) OR host of the bounty (`auth.uid() = bounties.host_id`). Both joiner and host need this for Realtime subscriptions to deliver UPDATE events when the host accepts.
- `bounty_requests` INSERT/UPDATE: blocked at table level; only via RPCs.
- `bounties` SELECT: public (anyone can browse the feed).
- `profiles` SELECT: own row + any profile referenced by a visible `room_presence` row (the projector needs names).

### Two QRs (explicit, named)

**Discover QR** — distributed widely (hallway cards, Discord, projector edge if helpful):
- Payload: `https://tabledrop.app/b/halftime-tabledrop`
- Action: lands on `/b/[slug]`. Shows bounty + tag form + "request to join."

**Check-in QR** — physical-only (taped to projector base, table, walls):
- Payload: `https://tabledrop.app/b/halftime-tabledrop?ci=<checkin_token>`
- Action: calls `check_in(slug, token)` RPC. Only succeeds if caller has an `accepted` request.

### Presence (sticky during bounty window)

Once `is_present = true`, joiner remains on `/room/[slug]` until `bounties.ends_at`. No 5-min heartbeat expiry. This fixes the previous contradiction where the Discord seed strategy would expire joiners before pitch open.

Optional client heartbeat from `/b/[slug]` updates `last_seen_at` for an "online dot" UI indicator only — does NOT remove rows.

### Realtime + RLS — the Single Highest-Risk Piece

Supabase Realtime publication is on `room_presence` filtered by `bounty_id`. Two known traps:

1. **Realtime respects SELECT RLS** — the anon-key projector subscription will NOT receive an UPDATE event for a row that transitions from `is_present = false` to `is_present = true` UNLESS the SELECT policy admits the new row state. Our SELECT policy admits `is_present = true`, so the post-flip row IS visible. But Postgres CDC behavior on the false→true transition specifically needs verification.
2. **REPLICA IDENTITY FULL** is required on `room_presence` for the Realtime change payload to include enough data for filter-by-bounty_id to work correctly.

**Mitigation: Friday evening 1-hour spike.** One dev builds the minimum POC (one table, one RLS policy, one anon-key subscription) and verifies the false→true UPDATE event fires. If it doesn't, fall back to: projector subscribes to ALL `room_presence` changes (no RLS filter on subscription) and filters client-side. This is a one-line change.

If neither works by Saturday noon: drop to 1-second polling instead of Realtime. Visually identical to the audience; technically less elegant. The demo doesn't suffer.

### Curation Queue UX (`/host/[slug]`)

Top section — **Pending requests** (Realtime-fed from `bounty_requests` filtered by bounty + status='pending'):

Each row:
- Avatar (32px) or initials placeholder
- Name (16px, bold)
- Tag chips (max 5, overflow shown as "+3")
- Created-at relative ("2m ago")
- `[Accept]` button — single-tap, calls `accept_request` RPC, row animates to "Accepted, awaiting check-in" section below
- `[Reject]` button — secondary button, calls `reject_request` RPC

Middle section — **Accepted, awaiting check-in** (collapsed by default, expand to use override):
- Same row layout
- Override button `[Mark arrived]` — calls `host_mark_arrived` RPC. Used for judges without phones or unwilling to scan.

Bottom section — **Present in room** (read-only, fed from same `room_presence` channel as projector):
- Confirms what's currently on the projector. Lets host verify counts.

Empty states for all three sections: "Doors open in N min" / "Queue empty" / "Nobody checked in yet."

### Realtime Failure Fallback

- Projector opens RT channel. If WebSocket disconnects >5s: silent fallback to 1s polling.
- Hidden hotkey `Cmd+R` on projector forces fetch.
- Sat evening: tether demo laptop to teammate's phone hotspot, test full flow on the alternate network.

### Error / Terminal States

`/b/[slug]` states:
- `bounty_closed` (now > ends_at): "This bounty has ended." + feed link.
- `bounty_full`: "Table's full. Here are other open bounties." + feed link.
- `awaiting_curation` (existing pending request): "You're in the queue. Host will accept soon."
- `accepted` (post-accept, pre-checkin): "You're in. Scan the Check-in QR at the table." + image of what the QR looks like + arrow pointing to where it lives.
- `rejected`: "Host declined this round." + feed link.
- `network failure`: exponential-backoff retry (1s, 2s, 4s) + toast.

`/room/[slug]` empty state: NOT "0 people." Curated empty state: *"Doors open at 3:15. The table is set."*

`/host/[slug]` empty state: per section above.

### Technical Demo Flow (engineer reference, 5 steps)

1. Judge scans **Discover QR** → `/b/halftime-tabledrop` → tag form + "request to join."
2. `request_to_join` RPC creates `bounty_requests` row in `pending`. Page transitions to "awaiting_curation" state.
3. Host (pre-authed) sees row in `/host/halftime-tabledrop`. Taps `[Accept]`. `accept_request` RPC fires. Row state → `accepted`.
4. Judge's `/b/halftime-tabledrop` page Realtime-receives the status change → transitions to "accepted" state → shows Check-in QR image and pointing instruction.
5. Judge scans **Check-in QR** at the table → `check_in` RPC → `is_present = true` → projector receives Realtime UPDATE → judge appears on `/room/halftime-tabledrop` within 2s.

## Approach B — Bailout Spec (if Saturday 4pm integration fails)

**Trigger:** Saturday 4pm checkpoint fails. The Realtime + RLS + check_in RPC chain is not green end-to-end.

**What survives from A:**

- `/b/[slug]` page (rename to a static, single-bounty page)
- `/room/[slug]` page (rename to projector, polling instead of Realtime)
- The Check-in QR concept (token validation, but via a single endpoint not an RPC)

**What gets cut:**

- All RPCs except `check_in_lite` (one endpoint).
- `bounty_requests` table — replaced by direct `INSERT` into `room_presence` on check-in scan.
- `/host/[slug]` curation queue → replaced by host paste-list (host has a Notion/Sheet of pre-approved names; check-in succeeds only if `p_name` is in the allowed-list env var).
- Magic link auth → replaced by HTTP basic auth on `/host` (which itself becomes a read-only "list" page, not an interactive queue).

**Approach B data model (minimal):**

```sql
room_presence (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  tags text[] default '{}',
  checked_in_at timestamptz default now()
);
```

One table. One endpoint. 1s polling on the projector. Curation enforced by the allowed-name list (env var or seed table).

**Time to ship B from A's progress:** ~2-3h (delete RPCs, rewire `/b/[slug]` to direct insert, switch projector to polling).

**What the pitch loses:** the visible curation animation (queue → accepted → present). Mitigation: host MANUALLY narrates ("I'm accepting this request — see, the name's appearing now"). The premise is told via voice, not UI.

This is the bailout. It's specified concretely because "drop to B" is a comforting fiction if B is not pre-imagined.

## What I noticed about how you think

- You described the pitch as the product before you described the product. *"We push in a table and everyone brings a chair, add some props in, and then we pitch like normal but we are seated and discussing the idea among ourselves but really loudly."* That sentence is the entire identity of what you're building. Codex caught it too. You're not building a bounty app. You're building software that makes a room feel intentionally assembled.

- You used the word *curate* twice in the same sentence as *create*. That's not generic product talk. That's editorial intent. Most apps in this space step BACK from curation because it doesn't scale. You stepped INTO it. That's your moat.

- You picked the rent-a-gf reference as inspiration without flinching at the PR risk. That's taste under pressure. Naming your actual inspiration directly instead of finding a sanitized analog is what separates a memorable pitch from a forgettable one.

- When Codex challenged your "no payment v1" premise, you didn't defend the original. You picked the cleanest accommodation: adopt the check-in gate, skip the bringing-an-artifact badge. That's a builder making a scope call, not a founder defending a position. For Halftime, the right move.

## Reviewer Concerns (carried forward, not blockers — engineers address inline)

The final spec review (v3) accepted the doc as a soft PASS at 7.5/10 with the load-bearing blockers (joiner auth, `bounty_requests` Realtime publication, host_id seeding) now fixed above. Remaining nitpicks the engineers should address as they build:

1. **`max_seats` race condition.** Concurrent accepts could exceed `max_seats`. Mitigation in `accept_request` RPC: `select count(*) ... where status='accepted' for update;` before insert. Not yet specified in the RPC body — apply when implementing.
2. **`bounties.checkin_token` is queryable via public SELECT.** Anyone with the anon key can query the token. Mitigation: create a view `bounties_public` that excludes the column, point the feed query at the view. Curation premise is still enforced by `check_in` RPC (token-only-without-acceptance returns `not_requested`), so this is defense-in-depth, not a critical fix.
3. **`profiles` SELECT policy with join through `room_presence`.** RLS subqueries are slow and easy to misconfigure. If the policy gets tricky, simpler alternative: make all `profiles` rows SELECT-public (no PII besides name + avatar) and trust premise #2 (curation gates who can join, so profile rows are inherently semi-public).
4. **Heartbeat RPC.** `last_seen_at` updates need their own RPC since table-level UPDATE is blocked. **Decision: cut `last_seen_at` and the online-dot UI entirely.** It's not in the demo success criteria and adds attack surface.
5. **Cut `host_mark_arrived` RPC + the "awaiting check-in" section in the host queue.** YAGNI for the hackathon. If a judge can't scan, the host scans on their behalf with their own phone (same effect, no new code).
6. **Cut `venue_lat`, `venue_lng` from `bounties`.** Geo-fallback was demoted to v1.1. The columns are unused.
7. **Two-of-three sections in host queue collapse to one.** Show pending only; accepted+present count goes in a header chip. Simpler UI, same demo signal.
8. **Discord+hallway throughput risk during pitch.** If 20 requests land in 60 seconds, host single-tap-accept may not keep up. Mitigation if observed in rehearsal: add a `[Accept all]` button. Spec it Saturday only if rehearsal shows congestion.
9. **Cmd+R hotkey on projector** conflicts with browser refresh. Either accept that (refresh re-mounts and re-fetches anyway) or pick a different combo like `Ctrl+Shift+R`.
10. **Approach B's allowed-name list:** use a `host_approved_names text[]` column on the bounty seed row, not env var. Editable via SQL mid-demo if needed. Picks the Notion/Sheet path off the table.

These are caught and survivable. They do NOT block starting the build.
